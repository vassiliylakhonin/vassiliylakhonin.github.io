Case Study: Nonprofit Proposal Decision Engine

Context

I rebuilt this skill around one principle: grant work should be treated as a submission decision system, not just a proposal drafting task. The implementation is published on ClawHub and designed for NGO teams under deadline pressure.

Problem

Many proposal workflows optimize for polished narrative but underperform on decision integrity: weak input discipline, unclear assumptions, poor donor-fit mapping, and missing verification steps.

Actions

  • Repositioned the skill as a decision engine with a required verdict: Go / Conditional Go / No-Go.
  • Added minimum input gating before full drafting (donor, geography, target group, scope, budget, timeline, partners).
  • Implemented explicit operating modes: concept, loi, full, review, donor-fit, express.
  • Standardized outputs into fixed sections: Decision Summary, FAHU split, artifacts, donor-fit matrix, evidence, readiness checklist.
  • Added confidence labels and anti-fabrication rules for evidence handling.
  • Enforced safety guardrails for budget realism, safeguarding, and compliance risk escalation.

What it does now

  • Produces donor-oriented proposal artifacts and a clear submission decision.
  • Separates facts, assumptions, hypotheses, and unknowns in every substantive output.
  • Builds donor-fit gap analysis with concrete fix actions.
  • Creates an Evidence Needed plan when supporting sources are missing.

Current outputs

  • Decision Summary (verdict + confidence + key reasons).
  • RBM/ToC structure and logframe-ready outputs.
  • MEAL mini-plan and budget logic summary.
  • Risk and safeguarding matrix.
  • Donor-fit matrix and submission readiness checklist.

Who it is for

  • NGO grant managers and proposal leads.
  • MEAL/M&E specialists.
  • Nonprofit consultants and bid teams.
  • Program directors who need defensible go/no-go decisions before submission.

Why this version is better

It improves decision quality and traceability under time pressure, while explicitly handling uncertainty and evidence gaps instead of masking them with confident prose.

Tech stack

  • OpenClaw runtime and orchestration.
  • ClawHub skill distribution.
  • Instruction-first architecture with clear contracts and guardrails.
